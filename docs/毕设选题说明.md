# 毕业设计选题说明

**选题名称：基于多智能体协同的智能运维（AIOps）故障诊断系统设计与实现**

---

## 一、这是一个什么项目

本项目构建一个由**大语言模型多智能体（Multi-Agent）协同**驱动的智能运维平台。系统模拟资深运维工程师的排障流程：当线上服务发生告警时，系统能够**自动接收告警 → 规划排查步骤 → 并行采集证据（监控指标、系统日志）→ 结合运维知识库进行根因分析 → 给出处理建议**，并在关键结论处引入人工审核确认。

平台采用 **Supervisor（调度）统一编排的三 Agent 架构**：

- **对话 Agent**：系统入口，负责理解用户意图、多轮对话，并将请求路由到合适的 Agent；
- **知识库 Agent**：基于检索增强生成（RAG），精准回答运维领域专业问题，杜绝模型臆造；
- **运维 Agent**：核心模块，采用"规划-执行-再规划"（Plan-Execute-Replan）模式自主完成故障诊断。

---

## 二、研究意义与价值

**现实痛点**：传统运维（OnCall 值班）在故障发生时，高度依赖工程师人工查看监控、检索日志、凭经验定位根因。这种模式存在三个突出问题：

1. **响应慢**：人工逐项排查耗时，故障影响范围随时间扩大；
2. **质量不稳定**：夜间值班、新人值班时排障质量明显下降；
3. **经验难沉淀**：资深工程师的排障经验停留在个人脑中，难以复用和传承。

**本课题的价值**：

- **技术前沿性**：将大模型 Agent 技术应用于运维领域是新兴的交叉方向，业界落地案例仍较少，具有探索价值；
- **实用性**：把"人工排障"升级为"AI 辅助/自动诊断"，并将专家经验固化进知识库，实现经验的可复用与可传承；
- **可信性探索**：通过"人在回路"机制保证 AI 决策在关键节点可控，回应了"AI 能否被信任用于生产运维"这一现实关切。

---

## 三、解决了什么问题

| 传统运维的问题 | 本系统的解决方式 |
|----------------|------------------|
| 排障靠人工，响应慢 | 自动接收告警，自主规划并并行采集诊断证据 |
| 模型直接问答易"胡编" | RAG 检索运维知识库，让诊断"有据可依" |
| 跨服务根因难定位 | 知识图谱（GraphRAG）构建服务依赖拓扑，多跳推理定位根因 |
| 固定脚本无法应对复杂故障 | Plan-Execute-Replan 模式，根据中间结果动态调整诊断计划 |
| AI 全自动执行有风险 | 人在回路（HITL），关键结论需人工审核确认 |
| "AI 到底有没有用"说不清 | 构建故障场景评测集，量化根因定位/路由/工具调用准确率 |

## 四、项目现状：已完成的工作基础

本课题**并非从零起步**，已在前期完成了一套功能完整的 **RAG 知识库问答系统**作为底座。在此基础上，毕设阶段的核心工作是融合运维场景、引入多 Agent 协同能力。已实现的模块如下：

**1. 文档处理流水线（已实现）**
- 多格式文档解析（PDF、Markdown、TXT），文本清洗与元数据抽取；
- 可配置的文本分块（Chunking）策略；
- 基于通义千问 Embedding 的向量化，存入 Milvus 向量数据库；
- 基于 Celery + Redis 的异步处理，支持大文件后台入库。

**2. 检索增强问答（已实现）**
- 向量检索 + 重排序（Rerank）两阶段召回，提升相关性；
- **多跳检索（Multi-hop）**：复杂问题自动拆解为子问题，分步检索再综合作答；
- 问题分析器：自动判断问题类型与是否需要多跳；
- 答案附带来源引用，并记录首字延迟、总延迟等性能指标。

**3. 记忆系统（已实现）**
- 短期记忆（多轮对话上下文，基于 Redis）；
- 长期记忆框架（预留向量化记忆检索接口）。

**4. 知识库与智能助手管理（已实现）**
- 多知识库管理、文档增删、公开/私有权限控制；
- 可配置智能助手（绑定知识库、系统提示词、模型参数、记忆策略）；
- 用户认证与基于角色的访问控制（RBAC）。

**5. 自动化评测（已实现）**
- 基于大模型的 RAG 评测器，从**忠实度、相关性、上下文精度、准确性**四个维度自动打分；
- 支持自动生成或上传问答测试集（Golden Dataset），批量评测并产出报告。

> 说明：以上 RAG 底座在毕设中将**重新定位为"运维知识底座"**（承载 Runbook、SOP、历史故障复盘），为运维 Agent 的诊断提供经验依据；现有的多跳检索、重排序、评测框架等能力将直接复用并扩展到运维场景。

## 五、毕设阶段待完成的核心工作

在上述 RAG 底座之上，毕设阶段重点完成以下**新增工作**：

1. 引入大模型函数调用（Function Calling）与 LangGraph 编排能力（**已完成技术验证**）；
2. 实现运维 Agent 的"规划-执行-再规划"诊断流程；
3. 实现 Supervisor 调度与三 Agent 协同路由；
4. 接入监控/日志工具服务（MCP），先模拟数据后接真实 Prometheus；
5. 构建基于 Neo4j 的服务依赖知识图谱，支持多跳根因推理；
6. 实现人在回路（HITL）的诊断审核与状态恢复；
7. 扩展评测体系至故障诊断场景，产出量化对比实验。

## 六、预期使用的关键技术

**1. 大语言模型与 Agent**
- 大模型：通义千问（Qwen）、DeepSeek（通过阿里云百炼统一接入）
- 函数调用（Function Calling）：让模型自主决定调用哪个工具
- LangGraph：多 Agent 编排框架，实现"规划-执行-再规划"的图式工作流

**2. 多 Agent 协同**
- Supervisor 调度模式：意图识别与动态路由
- 并行诊断：多个诊断任务并发执行（指标查询、日志查询、依赖分析）
- 人在回路（HITL）：基于状态持久化（Checkpointer）实现执行中断与恢复

**3. 检索增强与知识图谱**
- RAG：向量检索 + 重排序（Rerank），从运维知识库精准召回
- GraphRAG：基于 Neo4j 图数据库构建服务依赖关系，支持多跳根因推理

**4. 工具服务与数据源**
- MCP（Model Context Protocol）：将监控、日志等能力封装为标准化工具服务
- 监控数据：先用模拟数据验证诊断链路，后期接入真实 Prometheus 监控
